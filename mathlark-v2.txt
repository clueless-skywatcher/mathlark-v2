● Now I have the full picture. Here's what you need to code to write functions
  in Mathlark:

  ---
  What's Already Working

  User-defined functions work in .lk files via the LarkFileReader path. The
  syntax:

  <Add2> := {
      (a) := {
          Return a + 2;
      }
  }

  PrintLn(Add2(5));   // prints 7

  This works because LarkFileReadingVisitor has the scope + funcs map, and
  DefinedFunction.invoke() handles execution.

  What's Missing (What You Need to Code)

  1. If/Else Statements — The biggest gap

  Without conditionals, you can't write anything beyond straight-line code. No
  recursion, no branching.

  Grammar changes (both Lark.g4 and LarkFile.g4):
  // Add to codeBlock or as a statement type
  ifStmt
      : 'If' '(' expr ')' codeBlock ('Else' codeBlock)?
      ;

  Then update codeBlock to allow ifStmt:
  codeBlock
      : '{' (expr ';' | functionDef | ifStmt)* returnStmt? '}'
      ;

  Visitor changes (LarkFileReadingVisitor.java):
  @Override
  public IExpression visitIfStmt(IfStmtContext ctx) {
      IExpression condition = this.visit(ctx.expr()).evaluate();
      if (condition.equals(GlobalSymbols.TRUE)) {
          return this.visit(ctx.codeBlock(0));  // then-branch
      } else if (ctx.codeBlock().size() > 1) {
          return this.visit(ctx.codeBlock(1));  // else-branch
      }
      return GlobalSymbols.UNDEFINED;
  }

  2. While Loops

  For iterative algorithms (GCD, Newton's method, etc.).

  Grammar:
  whileStmt
      : 'While' '(' expr ')' codeBlock
      ;

  Visitor:
  @Override
  public IExpression visitWhileStmt(WhileStmtContext ctx) {
      IExpression result = GlobalSymbols.UNDEFINED;
      while (this.visit(ctx.expr()).evaluate().equals(GlobalSymbols.TRUE)) {
          result = this.visit(ctx.codeBlock());
      }
      return result;
  }

  3. REPL-defined functions don't work

  Currently Lark.g4 (the REPL grammar) parses functionDef but the evaluation is
  embedded in the grammar actions — it never flows through
  LarkFileReadingVisitor. The REPL creates FunctionCallExpression without
  scope/funcs context, so user-defined functions just return their string form
  (e.g., f1(1) → "f1(1)").

  To fix this, you'd need to either:
  - Give Lark.g4's REPL path access to a DefinedFunction map (stored globally
  like SymbolTables), or
  - Refactor the REPL to also use a visitor pattern instead of embedded grammar
  actions

  4. Logical operators (&&, ||, !)

  Needed for meaningful conditionals.

  Grammar addition:
  logicalAnd: expr '&&' expr ;
  logicalOr: expr '||' expr ;
  logicalNot: '!' expr ;

  5. Multiple overloads per function

  The grammar in LarkFile.g4 already supports this (functionDefs+), but Lark.g4
  only allows one (functionDefs without +). This lets you do pattern-matching
  style:

  <Factorial> := {
      (n) := {
          If (n == 0) { Return 1; }
          Return n * Factorial(n - 1);
      }
  }

  ---
  Recommended Implementation Order

  1. If/Else — unlocks recursion and branching (most value)
  2. Logical operators — makes conditionals useful
  3. While loops — iterative algorithms
  4. REPL function support — so you can test interactively without .lk files

  Would you like me to start implementing any of these?

  ---
  Lazy If Tutorial — Making Recursion Work in MathLark

  The Problem

  Right now, If(n == 0, 1, n * Factorial(n - 1)) eagerly evaluates all three 
  arguments before If even sees the condition. So Factorial(0) still computes
  Factorial(-1) → infinite recursion.

  The fix: wrap unevaluated branches in thunks — objects that hold a computation
   but don't run it until you ask. If receives thunks, checks the condition, and
   evaluates only the winning branch.

  The Architecture

  visitFunctionCall("If", [cond, then, else])
          │
          ├── Is "If" lazy? YES
          │
          ├── arg[0] (condition):  visit + evaluate normally →
  BooleanExpression(true)
          ├── arg[1] (then branch): wrap in ThunkExpression (NOT evaluated yet)
          └── arg[2] (else branch): wrap in ThunkExpression (NOT evaluated yet)
          │
          ▼
    IfFunc.mainEval([BooleanExpression, Thunk, Thunk])
          │
          ├── condition is True → call thunk[1].evaluate() → runs the
  computation
          └── thunk[2] is never evaluated → recursive branch never fires

  ---
  Step 1: Create ThunkExpression

  This is a deferred computation. It holds a Supplier (a lambda that produces a
  value) and caches the result so it's only computed once.

  Create src/main/java/io/mathlark/larkv2/expressions/ThunkExpression.java:

  package io.mathlark.larkv2.expressions;

  import java.util.function.Supplier;

  import io.mathlark.larkv2.expressions.math.NumericExpression;
  import io.mathlark.larkv2.symbols.GlobalSymbols;

  /**
   * A deferred computation. Wraps an unevaluated expression that only
   * runs when .evaluate() is called. Used by lazy functions like If()
   * to avoid evaluating branches that shouldn't execute.
   *
   * The result is cached — calling evaluate() twice returns the same
   * value without recomputing.
   */
  public class ThunkExpression implements IExpression {
      // The deferred computation. This lambda captures the visitor
      // and parse tree context, so calling .get() will visit/evaluate
      // the expression on demand.
      private Supplier<IExpression> supplier;

      // Once evaluated, the result is cached here so we never
      // recompute. This is important because evaluate() may be
      // called from multiple places (checkArgs, mainEval, etc.)
      private IExpression cached;

      public ThunkExpression(Supplier<IExpression> supplier) {
          this.supplier = supplier;
      }

      @Override
      public IExpression evaluate() {
          // Lazy evaluation with caching: only compute once
          if (cached == null) {
              cached = supplier.get();
          }
          return cached;
      }

      // All arithmetic operations force evaluation first.
      // If a thunk ends up in an arithmetic context (e.g. thunk + 1),
      // it evaluates itself and delegates to the real expression.

      @Override
      public IExpression add(IExpression other) {
          return evaluate().add(other);
      }

      @Override
      public IExpression sub(IExpression other) {
          return evaluate().sub(other);
      }

      @Override
      public IExpression mul(IExpression other) {
          return evaluate().mul(other);
      }

      @Override
      public IExpression div(IExpression other) {
          return evaluate().div(other);
      }

      @Override
      public IExpression mod(IExpression other) {
          return evaluate().mod(other);
      }

      @Override
      public IExpression pow(IExpression other) {
          return evaluate().pow(other);
      }

      @Override
      public IExpression negate() {
          return evaluate().negate();
      }

      @Override
      public StringExpression head() {
          return new StringExpression("Thunk");
      }

      @Override
      public Object val() {
          return evaluate().val();
      }

      @Override
      public String inputForm() {
          return evaluate().inputForm();
      }

      @Override
      public String toString() {
          return evaluate().toString();
      }

      @Override
      public IExpression length() {
          return evaluate().length();
      }
  }

  Key idea: the Supplier<IExpression> is a lambda like () -> 
  visitor.visit(exprCtx).evaluate(). It captures the visitor and parse tree
  node. Calling supplier.get() runs the visitor on that subtree — but only when
  we choose to.

          if (ctx.actualParams() != null) {
              List<IExpression> params = new ArrayList<>();

              // Check if this is a lazy function (like If).
              // Lazy functions receive ThunkExpressions — unevaluated

              boolean lazy = UniversalFunctionRegistry.isLazy(funcName);

              for (ExprContext expr: ctx.actualParams().expr()) {

                      // Function references are always passed as names
                      params.add(new StringExpression(expr.getText()));
                  }
                  else if (lazy) {
                      // Lazy path: wrap in a thunk that captures the
                      // current visitor and this parse tree node. The
                      // expression won't be evaluated until the function
                      // explicitly calls .evaluate() on it.
                      final ExprContext capturedExpr = expr;
                      params.add(new ThunkExpression(

                      ));
                  }
                  else {
                      // Eager path (default): evaluate the argument now
                      params.add(this.visit(expr).evaluate());
                  }
              }
              return new FunctionCallExpression(funcName, params, scope, funcs);
          }
          return new FunctionCallExpression(funcName, List.of(), scope, funcs);
      }

  this.visit(capturedExpr).evaluate() captures this (the visitor) and the parse
  tree node, forming a closure that can be invoked later.

          IExpression condition = exprs[0].evaluate();

              // computation never runs. This is what prevents
              // infinite recursion.
              return exprs[1].evaluate();

              return exprs[2].evaluate();

          // the condition evaluated to check its type.
          if (!FunctionUtils.isInstanceOf(exprs[0], BooleanExpression.class)) {

  package io.mathlark.larkv2.symbols;

  import java.util.HashMap;
  import java.util.Map;

  /**
   * Global registry for user-defined functions loaded from .lk files.
   *
   * Functions are stored with name-mangled keys: "FuncName" + arity.
   * For example, Abs(x) is stored as "Abs1", Max(a, b) as "Max2".
   * This is the same mangling scheme used by DefinedFunction throughout
   * the codebase.
   *
   * This table is populated at startup when stdlib .lk files are loaded,
   * and consulted by FunctionCallExpression when the REPL encounters
   * a function call that isn't a built-in.
   */
  public class FunctionTable {
      private static final Map<String, DefinedFunction> functions = new
  HashMap<>();

      /**
       * Store all functions defined in a .lk file into the global table.
       * Called by LarkFileReader after executing a file.
       */
      public static void registerAll(Map<String, DefinedFunction> funcs) {
          functions.putAll(funcs);
      }

      /**
       * Look up a user-defined function by name and arity.
       * Returns null if not found.
       */
      public static DefinedFunction get(String name, int arity) {
          return functions.get(String.format("%s%d", name, arity));
      }

      /**
       * Check if a user-defined function exists with the given name
       * and arity.
       */
      public static boolean has(String name, int arity) {
          return functions.containsKey(String.format("%s%d", name, arity));
      }

      /**
       * Returns the full function map. Needed when invoking a
       * DefinedFunction, because it may call other user-defined
       * functions internally (e.g., Max calls If).
       */
      public static Map<String, DefinedFunction> getAll() {
          return functions;
      }
  }

              // from the REPL and other .lk files.

          }

       * instead of CharStreams.fromFileName(), because resources inside
       * a JAR aren't filesystem files.
       */
      public static void loadResource(String resourcePath) {
          try {
              InputStream stream = LarkFileReader.class
                  .getClassLoader()
                  .getResourceAsStream(resourcePath);

              if (stream == null) {
                  System.err.println("Could not find resource: " +
  resourcePath);

❯ Which functions should I now implement in Lark and which functions in JVM?    
How do I decide which function to implement where?                              

● Decision Framework

  Implement in Java when the function needs to:

  1. Access JVM/internal types — it reaches into IExpression internals, casts to
   specific classes, or constructs expressions that only Java can build
  2. Break normal evaluation rules — Return() throws an exception, If() needs
  lazy args, Map() invokes functions by name through the registry
  3. Interact with the outside world — I/O, file reading, system calls (PrintLn,
   Exit)
  4. Manipulate the runtime — scope, symbol tables, clearing state (Clear,
  ClearAll, DefinedVars)
  5. Perform operations on opaque data structures — list internals (First, Last,
   Rest, Append, ElementAt), dict internals (DictKeys, DictVal), permutation
  internals

  Implement in MathLark when the function is:

  1. Pure arithmetic/logic composed of existing operations — Abs, Sign, Max,
  Min, Clamp
  2. A mathematical formula — anything that's just +, -, *, /, comparisons, and
  calls to other functions
  3. A composition of existing functions — Clamp(x, lo, hi) calls Min and Max

❯ Ok, I want a memoized implementation of functions like Factorial and

● You can't — not yet. Here's why and what's missing:

  What Memoization Needs

  A memoized Fibonacci looks like this conceptually:

  cache := Dict({});

  <Fib> := {
      (n) := {
          If(DictHasKey(cache, n),
              Return(DictVal(cache, n)),
              ...
          );
          result := If(n <= 1, n, Fib(n - 1) + Fib(n - 2));
          cache := DictPut(cache, n, result);
          Return(result);
      }
  }

  This requires four things your language doesn't have yet:

  1. Mutable state across function calls

  DefinedFunction.invoke() creates a fresh SymbolScope for every call with
  isFunc = true, which blocks lookup into parent scopes. The cache variable
  defined outside the function body is invisible inside it. Even if you could
  read it, reassigning cache inside the function wouldn't update the outer
  variable — SymbolScope.assign with isFunc = true restricts the lookup chain.

  2. Dict mutation functions

  You have Dict, DictKeys, DictVal, DictVals — all read-only. You're missing:
  - DictHasKey(dict, key) — check if a key exists
  - DictPut(dict, key, value) — return a new dict with the entry added

  These are Java functions (they need to construct/inspect DictExpression
  internals).

  3. Multiple statements before Return

  Your codeBlock supports (expr ';')*, so sequential statements work. But the
  scoping issue in point 1 means intermediate assignments inside a function
  can't modify outer state.

  4. Lazy If with side effects

  If is lazy now, which is great. But memoization needs the chosen branch to
  both return a value and update the cache. That requires either multiple
  statements in a branch (code blocks as If arguments) or a helper function that
   does both.

  Two Paths Forward

  Path A: Global mutable table (simplest)

  Add a single Java-level built-in — a global memo table:

  - Memo(funcName, key) — look up cached result, returns Undefined if miss
  - MemoSet(funcName, key, value) — store a result, returns value

  Then in MathLark:

  <Fib> := {
      (n) := {
          Return(If(Memo('Fib', n) != Undefined,
              Memo('Fib', n),
              MemoSet('Fib', n, If(n <= 1, n, Fib(n - 1) + Fib(n - 2)))
          ));
      }
  }

  This requires two new Java functions (they manage a static Map<String, 
  Map<IExpression, IExpression>>) and zero language changes.

  Path B: Fix scoping to allow closures (proper but bigger)

  Change SymbolScope.resolve() so that function scopes can read (but not write)
  variables from enclosing scopes. Then a Dict defined outside the function is
  visible inside it. You'd also need DictHasKey and DictPut in Java. This is
  more work but gives you real closures.

  My Recommendation

  Path A. Two small Java functions (Memo and MemoSet) give you memoization for
  any recursive function with zero language changes. Path B is worth doing
  eventually, but it's a separate feature (closures) that shouldn't block
  memoization.

❯ What does the Module[] function do in Mathematica?                            

● Module creates a block with local variables. It's Mathematica's scoping
  construct.

  Module[{x, y = 5}, body]

  - {x, y = 5} — declares local variables, optionally with initial values
  - body — an expression that can use those variables
  - Variables are truly local — they don't leak out or shadow outer bindings
  permanently

  Example

  Module[{x = 3, y = 4},
    x^2 + y^2
  ]
  (* returns 25 *)

  Key behaviors

  - Unique symbols: Mathematica internally renames locals to x$123 etc., so they
   never collide with global x
  - Mutable: You can reassign locals within the body (x = x + 1)
  - Return value: The value of the last expression in the body
  - Nested: Modules can nest, each with its own scope

  Compared to what you have in MathLark

  This is essentially what your codeBlock + SymbolScope already does inside
  function bodies. The difference is Mathematica exposes it as a standalone
  function you can use anywhere, not just inside function definitions. If you
  wanted Module in MathLark, it would be another lazy/special function that:

  1. Creates a new SymbolScope
  2. Binds the declared variables
  3. Evaluates the body in that scope
  4. Returns the result

  Changes

  Grammar (LarkFile.g4 and Lark.g4)

  Added a typedParam rule that makes type annotations optional:

  typedParam
      : name=IDENTIFIER (':' type=IDENTIFIER)?
      ;

  functionDefs[String funcName]
      : '(' (typedParam (',' typedParam)*)? ')' ':=' codeBlock
      ;

  (x) still works (untyped). (x: Numeric) now also parses.

  Visitor (LarkFileReadingVisitor.java)

  visitFunctionDefs iterates over typedParam contexts instead of raw Token args.
   Extracts both param.name and param.type (null if absent).

  DefinedFunction.java

  - New field List<String> paramTypes — parallel to paramNames, with null
  entries for untyped params
  - invoke() checks types before binding params — if a typed param doesn't
  match, returns Undefined immediately
  - matchesType() compares against expr.head().toString() — the same type names
  used by Head() in the REPL
  - "Numeric" is a special alias matching both "Integer" and "Decimal"
